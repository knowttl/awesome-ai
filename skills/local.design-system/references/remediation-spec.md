# design-remediation.md Specification — Drift Report

Read this when executing Workflow step "Generate all files" in **codebase-audit** or **hybrid** mode. Not generated in reference-only mode.

## Purpose

This file is the hand-off artifact for a *different* AI agent (or the same agent in a later session) whose job is to bring the existing UI into line with the design system. It names **what** is inconsistent and **what the standard should be** — not where. Per project decision, this file stays at **category/component level, with no file paths or line numbers**: the remediation agent is expected to re-scan the codebase itself to locate every instance of a flagged pattern before fixing it. Do not shortcut this by guessing at locations.

```markdown
<!-- Generated by design-system skill | [YYYY-MM-DD] | Sources: codebase scan of [path(s)] -->
```

#### Drift Details
Group with H3 subsections matching `design.md` / `design-components.md` categories (Color, Typography, Spacing, Shape, Elevation, Components, Layout, Motion). One entry per distinct issue:

```markdown
### Card corner radius

**Found:** 3 competing values in use — 8px (majority), 4px, and 12px.
**Standard:** `radius.medium` (12px) is the defined token for Card-level containers — see design.md.
**Severity:** Medium — visually inconsistent but not accessibility-breaking.
**Recommendation:** Replace the 8px and 4px variants with `radius.medium`. If a component intentionally needs a different radius, it should be reclassified (not Card) or the token set extended — flag to the user rather than silently adding a new radius value.
```

**Severity scale** (state which applies and why, do not just label it):
- **High** — breaks accessibility (contrast, touch target, focus visibility) or creates jarring visual inconsistency on a high-traffic component.
- **Medium** — inconsistent but not accessibility-breaking; noticeable on close inspection.
- **Low** — minor inconsistency, low visual impact, or affects a rarely-used component.

## Writing up the chosen standard

The canonical value for each category is decided in Workflow step "Derive or confirm the system" (dominant pattern in Codebase-audit mode, reference-derived in Hybrid mode) — don't re-derive it here, just report it. Always state the frequency count or percentage for the values found, so a human or agent can sanity-check the call (e.g., "8px used in 71% of Card instances scanned").

## General Rules

- Every entry must state a concrete **Standard** to converge on — never flag an issue without naming the fix.
- Do not invent drift that isn't there: only list a category if 2+ distinct values are serving the same semantic purpose, or a value contradicts an explicit reference/token.
- If the scan found zero drift, state that plainly in the summary rather than omitting the file.
