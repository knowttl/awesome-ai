# design-preview.html Specification — Living Style Guide

Read this when executing Workflow step "Generate all files" for `DESIGN/design-preview.html`.

## Purpose

A single self-contained HTML page the user can open in a browser to see and click through the actual design system, then ask the agent for changes. It is a **rendering of the token/spec files**, not an independent design — every color, size, and radius on the page must come from a value already written in `design.md` / `design-components.md` / `design-guidelines.md`. Never invent a value here that isn't in those files.

## Constraints

- **Single file, no external dependencies.** Inline `<style>` and `<script>` only — no CDN links, no build step, no network requests. It must open correctly via `file://`.
- **Real markup, not lookalikes.** Use semantic elements matching each documented component (`<button>`, `<input>`, `<nav>`, etc.), not `<div>`s styled to resemble them — so states like `:disabled`, `:focus-visible`, and `:hover` work natively and the page doubles as something inspectable/testable.
- **Interaction states are live**, not static swatches: implement Rest/Hover/Focus/Pressed/Disabled with real CSS (`:hover`, `:focus-visible`, `:active`, `[disabled]`) wherever design-components.md documents them.

```html
<!-- Generated by design-system skill | [YYYY-MM-DD] | Sources: design.md, design-components.md, design-guidelines.md -->
<!doctype html>
<html lang="en" data-theme="light">
<head>
  <meta charset="utf-8">
  <title>[System Name] — Design Preview</title>
  <style>/* tokens + component CSS, see below */</style>
</head>
<body>
  <!-- sections, see Page Structure -->
  <script>/* theme toggle only, see Theme Toggle */</script>
</body>
</html>
```

## Token → CSS mapping

Generate a `:root { }` block whose custom property names match the **Design Tokens** table in `design.md` verbatim (e.g. `color.primary` → `--color-primary`) so every rule in the page is traceable back to a named row in that table. If dark mode is documented, add a second block keyed to the same theming strategy `design.md` describes (`[data-theme="dark"] { }`, or `@media (prefers-color-scheme: dark)` if that's what's documented — match it, don't pick a different mechanism). Component CSS must reference these custom properties (`background: var(--color-primary)`), never hardcode a hex/px value that duplicates a token.

## Theme Toggle

Only include if `design.md` documents dark mode support. A single button in the page header that flips `data-theme` between `light` and `dark` on `<html>`. This is the only JavaScript the page needs — no other interactivity is required.

## Page Structure

In order:

1. **Header** — system name, 1-line description, theme toggle button (if applicable).
2. **Colors** — every role from `design.md`'s Colors table as a swatch: filled block in the color, label with role name + hex, and a text sample in the paired `on-*` color to visually confirm contrast.
3. **Typography** — every scale level from `design.md`'s Typography table, each rendered as an actual text sample at its real size/weight/line-height, labeled with the style name.
4. **Spacing** — every token from the Spacing table, rendered as a sized box (width = token value) labeled with the token name and value, so the scale is visually comparable.
5. **Shape & Elevation** — a box per Shape token showing its radius, and a box per Elevation level showing its shadow, each labeled.
6. **Components** — one subsection per category, matching `design-components.md`'s H2 headings exactly (Actions, Input, Navigation, Containment, Data Display, Feedback). Render **every** component documented there, including its documented variants and states (e.g. every Button type × its states). Skip nothing — if a section is empty because the design system genuinely has no components in that category, omit the heading rather than rendering a placeholder.
7. **Footer** — generation date and source files, matching the header comment.

## Accessibility

Since interaction states are real CSS, verify (by inspection, not guessing) that focus outlines are actually visible against every background they appear on — this is the one place the design system's contrast rules are directly checkable.

## Keeping it in sync

The preview is generated **from** the doc files, never the other way around. If the user reviews the page and asks for a change, update the source (`design.md` for tokens, `design-components.md` for component specs, `design-guidelines.md` for rules) first, then regenerate this file from the updated source — see the Workflow's "Reviewing & Editing the Preview" section.
